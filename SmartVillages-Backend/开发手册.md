# 智慧乡村后端开发手册（脚手架流程版）

> 目标：你下次做新模块/新项目时，照着这份手册就能从 0 到 1 快速搭出“可运行、可鉴权、可文档、可闭环”的后端脚手架。

## 0. 项目定位与原则

- **先闭环再优化**：每个模块必须能“创建 → 列表 → 详情 → 更新/删除 → 权限可控 → 可在 Knife4j 验证”。
- **三角色职责固定**：
  - 管理员：账号管理
  - 村干部：业务执行（村务治理）
  - 村民：前台浏览与反馈
- **配置安全**：所有密码/密钥/AccessKey 用环境变量或本地 profile；不要写死提交进仓库。

## 1. 工程脚手架（一次性搭好）

### 1.1 Maven 多模块结构

- `common`：基础设施（统一返回、异常、JWT、Security、Redis 工具、通用配置）
- 业务模块：`auth/admin/announcement/interaction/management/media/feature`
- `service`：唯一启动模块（聚合所有业务模块）

### 1.2 启动与打包命令

```bash
./mvnw -pl service -am spring-boot:run
./mvnw -pl service -am clean package
```

## 2. 运行环境脚手架（先能跑起来）

### 2.1 application.yml（环境变量优先）

- DB：`DB_URL/DB_USERNAME/DB_PASSWORD`
- Redis：`REDIS_HOST/REDIS_PORT/REDIS_PASSWORD/REDIS_DB`
- JWT：`JWT_SECRET/JWT_EXPIRE_MS`
- OSS（可选）：`ALIYUN_OSS_*`

示例模板：`service/src/main/resources/application-example.yml`

### 2.2 SQL 脚手架

目录：`service/src/main/resources/sql/`

- `auth.sql`：认证表 + 示例账号
- `admin.sql`：管理员表
- 其它业务模块各自独立 SQL（announcement/interaction/management/media/feature）

## 3. 鉴权脚手架（固定套路）

### 3.1 登录签发 JWT

- 接口：`POST /auth/login`
- 登录成功返回：`JwtResponse(token + 基本用户信息)`
- token 放请求头：`token: <JWT>`

### 3.2 过滤器与上下文

在 `common`：

- `JwtSecurityFilter`：解析 token → 写入 `SecurityContext`（`ROLE_*`）→ 同时把 `authId/username/role` 写入 request attribute
- `LoginUserContext`：从 request 读取当前登录用户（业务模块取当前人都走它）

### 3.3 SecurityConfig（权限靠它兜底）

固定做法：

- 文档/登录接口：`permitAll`
- 管理员路径：`/admin/**` → `ROLE_ADMIN`
- 村干部路径：`/cadre/**` → `ROLE_CADRE`
- 村民路径：`/villager/**` → `ROLE_VILLAGER`
- 其它：`anyRequest().authenticated()`

**新增 Controller 前缀时第一时间同步改 `SecurityConfig`**，否则最常见就是 401/403。

## 4. 新模块开发脚手架（你以后照这个写）

> 适用于：公告/台账/工单/风采/留言等所有 CRUD + 状态流转类模块。

### 4.1 Step 1：建表（SQL）

在 `service/src/main/resources/sql/<module>.sql`：

- 主键：`INT AUTO_INCREMENT`
- 逻辑删除：`deleted TINYINT DEFAULT 0`
- 时间字段：`create_time/update_time`
- 索引：按查询维度建（status/type/create_time/user_id）

### 4.2 Step 2：Entity（MyBatis-Plus）

在业务模块 `entity/`：

- `@TableName`
- `@TableId(IdType.AUTO)`
- `@TableLogic` + `deleted`
- `@TableField(fill = FieldFill.INSERT/INSERT_UPDATE)`：createTime/updateTime

### 4.3 Step 3：DTO / VO（接口层契约）

在业务模块：

- `dto/*CreateDTO`：创建入参
- `dto/*UpdateDTO`：更新入参
- 若有状态流转：额外 `dto/*AuditDTO` / `*StatusDTO`
- `vo/*SimpleVO`：列表展示字段（少）
- `vo/*DetailVO`：详情字段（全）

### 4.4 Step 4：Mapper

- `extends BaseMapper<Entity>`
- `@Mapper`

### 4.5 Step 5：Service / ServiceImpl（业务规则落地）

Service 接口建议统一 5 件套：

- `create(...)`
- `page(...)`（分页 + 条件筛选）
- `detail(id)`
- `update(id, dto)`
- `delete(id)`

实现层套路：

- 分页：`LambdaQueryWrapper` + `Page<>(current,size)` + `convert(entity -> vo)`
- 详情：`getById` → copy 到 DetailVO
- 更新/删除：更新成功后淘汰缓存（如果有缓存）

### 4.6 Step 6：Controller（路径与角色一致）

建议路径风格：

- **管理端（干部）**：`/cadre/<resource>`（增删改查、审核、统计）
- **公共端（村民/游客）**：`/<resource>`（只读已发布/已生效数据）
- 统一返回：`Result.success(...)`

### 4.7 Step 7：缓存（可选但加分）

适用于“详情高频读”：

- key 约定：`<biz_prefix>:detail:<id>`
- 读取：先查缓存，没有再查 DB 并 set
- 写入（更新/删除/发布）：delete 缓存

工具：`RedisJsonCacheTool`、`CacheKeyUtils`

### 4.8 Step 8：验收清单（必须跑通）

- Knife4j 能看到接口 + 字段
- 401：无 token 访问受保护接口
- 403：低权限访问高权限路径
- CRUD 全通：创建 → 列表 → 详情 → 更新 → 删除
- 若有状态流转：状态从 A→B→C 跑一遍

## 5. 本项目复盘（你这次写项目的路线）

你这次的路线可以复用成模板：

1. **先做鉴权（JWT + SecurityConfig）** → 所有模块都能套用
2. **统一 Result/Exception/ErrorCode** → 保证接口一致性
3. **按业务域分模块** → announcement/interaction/management/media/feature…
4. **每个模块按“SQL→Entity→DTO/VO→Mapper→Service→Controller→权限→验收”** 顺序闭环
5. **高频详情再加 Redis 缓存**（读缓存 + 写淘汰）
