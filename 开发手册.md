2.**配置与数据库脚本**

- **唯一运行时配置**：`service/src/main/resources/application.yml`（数据源、MyBatis-Plus、Knife4j 等；OpenAPI 由 Knife4j starter 自带 SpringDoc，一般不必再单独维护一套 springdoc 依赖）。
- **建表脚本**：放在 `service/src/main/resources/sql/`，按模块分文件（如 `auth.sql`、`management.sql`）；改表结构后同步改对应 **Entity** 与脚本，并在 **README** 里「数据表」一行用 **Markdown 相对链接** 指到该 sql，方便点进仓库即见 DDL。
- 本地库名、账号密码与生产环境**不要写进仓库**时可改用环境变量或 Spring Profile（`application-dev.yml` 等），本仓库若暂用明文仅适合个人开发机。

3.**接口文档（Knife4j / OpenAPI）**

- 依赖在 **`common`**：`knife4j-openapi3-jakarta-spring-boot-starter`（已带 springdoc，不要在 `common` 再单独引一套 Swagger UI）。
- **`application.yml`**：`springdoc.swagger-ui.enabled: false`（关掉默认 Swagger UI），`knife4j.enable: true`；**不必**再写 `springdoc.api-docs.path`（默认即为 `/v3/api-docs`，Knife4j 与 JWT 放行逻辑依赖此路径）。浏览器访问 **`http://localhost:<端口>/doc.html`**。
- **全局文档标题**：在 **`OpenApiConfig`**（`common`）里配置 `OpenAPI` Bean；**不要**再使用名为 `SwaggerConfig` 的旧类名，以免与 Swagger2 概念混淆。
- Controller 上可使用 `@Tag`、`@Operation` 等 **OpenAPI3** 注解（`io.swagger.v3.oas.annotations`）。

3.1 **统一错误码、统一返回与全局异常**（与 README「统一返回与错误约定」表一致）

- **`ErrorCode`**（`com.backend.common.enums.ErrorCode`）  
  - 按 **码段** 维护：`1xxx` 用户与认证，`2xxx` 权限与角色，`3xxx` 文件与配置，`4xxx` 通用业务 + 村务域扩展（如 `41xx` 公告/村务事项/留言/媒体），`5xxx` 系统，`6xxx` 接口/HTTP 语义。  
  - **原则**：同一业务含义对应 **唯一** `code`；新增枚举项时在所属段内 **递增编号**，避免多个常量共用同一数字。  
  - 使用：`ErrorCode.XXX.getCode()`、`getMessage()`（Lombok `@Getter`）。

- **`Result`**（`com.backend.common.result.Result`）  
  - 约定所有接口返回 **同一 JSON 形状**（至少包含业务 `code`、提示 `message`、载荷 `data`；字段名以实现为准）。  
  - Controller 成功时 `Result.ok(data)`，失败时 `Result.fail(ErrorCode.XXX)` 或抛业务异常由全局处理。

- **`GlobalExceptionHandler`**（`@RestControllerAdvice`）  
  - 集中处理：**参数校验**（如 `MethodArgumentNotValidException`）、**业务异常**（`BusinessException`）、**静态资源 404**（`NoResourceFoundException`）、**405**、**未捕获异常** → `SYSTEM_ERROR` 等。**注意**：请求在 **`JwtAuthenticationFilter`** 中因缺 token/非法 JWT 被挡时，多为 **HTTP 401**，**不经过** MVC 与本处理器，也通常 **无 `Result` 体**；若日后改为在 Controller 内解析 JWT，再考虑对 `JwtException` 做 `Result` 映射。  
  - 业务层可定义 **携带 `ErrorCode` 的自定义异常**，在处理器里统一转 `Result`，业务方法内 **不必** 处处 `try-catch`（与过滤器、调用链是否进入 MVC 有关，见实现）。  
  - 新增异常类型时：**先** 在 `ErrorCode` 中增加或复用枚举，**再** 在处理器中增加 `@ExceptionHandler` 分支。

4.**可复用项目架构**：Maven 多模块 + 根包 `com.backend.*`（单 Spring Boot 进程，由 `service` 启动）。复制到新项目时保留「模块边界」与下面「包—职责」对应关系即可。

**依赖与职责约定**

| 位置 | Maven 依赖 | 主要内容 |
|------|------------|----------|
| 父工程 `smartVillages/` | 无业务依赖 | `pom.xml`：`<modules>`、`dependencyManagement`（第三方版本钉死） |
| `common` | 不写对业务模块的依赖 | Web、MyBatis-Plus、Knife4j(OpenAPI3/Jakarta)、JWT；`enums`（ErrorCode）、`config`、`exception`、`filter`、`result`、`utils` |
| 各业务子模块（如 `auth`、`admin`、`management`…） | 仅依赖 `common`（及 Lombok 等） | 领域代码，包名 `com.backend.<模块名>` |
| `service` | 依赖 `common` + 所有业务模块 + MySQL 驱动 | `SmartVillagesApplication`、`application.yml`、`resources/sql/`、集成测试 |

**各模块 `pom.xml` 依赖怎么写（可直接套模板）**

- 版本号：业务相关的第三方（MyBatis-Plus、Knife4j、jjwt 等）放在**父工程**的 `<dependencyManagement>` 里统一写 `<version>`；子模块里**不写版本**，只写 `<groupId>` + `<artifactId>`。
- 内部模块：一律 `<version>${project.version}</version>`，与父工程 `0.0.1-SNAPSHOT` 对齐。

**common**（公共能力，不要依赖任何业务 `artifactId`）

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-webmvc</artifactId>
    </dependency>
    <dependency>
        <groupId>com.baomidou</groupId>
        <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>com.github.xiaoymin</groupId>
        <artifactId>knife4j-openapi3-jakarta-spring-boot-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>io.jsonwebtoken</groupId>
        <artifactId>jjwt</artifactId>
    </dependency>
    <dependency>
        <groupId>javax.xml.bind</groupId>
        <artifactId>jaxb-api</artifactId>
    </dependency>
    <dependency>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
        <optional>true</optional>
    </dependency>
</dependencies>
```

**业务子模块**（以 `auth` 为例；`admin`、`announcement` 等只改 `<artifactId>auth</artifactId>` 与目录名）

```xml
<dependencies>
    <dependency>
        <groupId>com.backend</groupId>
        <artifactId>common</artifactId>
        <version>${project.version}</version>
    </dependency>
    <dependency>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
        <optional>true</optional>
    </dependency>
</dependencies>
```

**service**（启动模块）：`<dependencies>` 里聚合 **common**、**当前全部业务子模块**（各一条 `com.backend` + `${project.version}`），再加 `spring-boot-starter-webmvc`、`mysql-connector-j`（`runtime`）、`spring-boot-starter-test`（`test`）；`<build>` 里配置 `spring-boot-maven-plugin`。**完整写法直接看仓库里的 `service/pom.xml`，不在手册里重复粘贴，避免以后要改两处。**

新增业务模块时：父 `pom` 的 `<modules>` 增一行，**`service/pom.xml` 的 `<dependencies>` 增一条同模块**，再根目录 `mvn install`。

5.分完包，就对子模块的依赖版本进行控制，下面是必要的父依赖填写信息

```
<parent>
    <groupId>com.backend</groupId>
    <artifactId>smartVillages</artifactId>
    <version>0.0.1-SNAPSHOT</version>
    <relativePath>../pom.xml</relativePath>
</parent>
```

6. **`service/pom.xml`** 必须带上所有业务模块依赖，漏了哪个，运行时就不会加载那个包下的 Bean。若只改了 `common`，建议根目录 `mvn install`（或至少 `install common`）再运行/打包。

7.根目录不要再保留旧的 `src/main`；`application.yml` 与 `sql` 只放在 `service/src/main/resources/`。

8.本地运行与打包

- **在仓库根目录执行**（需已安装 JDK 17、本机 MySQL 与 `application.yml` 中库名/账号一致）：
  ```bash
  mvn -pl service -am spring-boot:run
  ```
  或在 IDE 中 **运行** `service` 模块下的 `com.backend.SmartVillagesApplication`。
- **打可执行包**（产物在 `service/target/*.jar`）：
  ```bash
  mvn clean package -DskipTests -pl service -am
  java -jar service/target/service-0.0.1-SNAPSHOT.jar
  ```
- 多模块下若刚改过 **`common`** 或其它被依赖模块，其他模块仍用旧 class，请先 **`mvn install`**（根目录执行一次最省事）。
